# DCS (Decentralized Communication System)

## Описание проекта

DCS — это децентрализованная система связи для локальных сетей, работающий без центрального сервера. Приложение позволяет пользователям обмениваться текстовыми сообщениями и голосовыми данными напрямую между устройствами в одной сети или через сложные сетевые конфигурации. Проект разработан с использованием Rust для бэкенда и веб-технологий для фронтенда, упакованных в нативное приложение через Tauri.

Основная цель проекта — обеспечение связи в условиях отсутствия интернета, при обрыве внешней сети или в изолированных сегментах корпоративной инфраструктуры.

## Архитектура системы

Система построена по принципу P2P (Peer-to-Peer). Каждое запущенное приложение является одновременно клиентом и сервером. Нет главного узла, координирующего трафик.

### Стек технологий

- **Бэкенд:** Rust. Обеспечивает высокую производительность, безопасность памяти и работу с низкоуровневыми сетевыми сокетами.
- **Фронтенд:** HTML5, CSS3, JavaScript (ES6+). Интерфейс рендерится через движок WebView.
- **Фреймворк:** Tauri. Связывает Rust и веб-интерфейс, позволяя создавать легкие кроссплатформенные исполняемые файлы.
- **Криптография:** Библиотека Sodiumoxide (обертка над libsodium). Используется симметричное шифрование для защиты содержимого сообщений.
- **Сетевые протоколы:** UDP (для обнаружения и голоса), TCP (для гарантированной доставки текста).

## Сетевые алгоритмы и протоколы

### 1. Обнаружение узлов (Discovery Protocol)

Для поиска других участников в сети используется механизм широковещательной рассылки (Broadcast).

- **Протокол:** UDP.
- **Порт:** 5000.
- **Алгоритм:**
    1. При запуске приложение создает UDP-сокет и привязывается ко всем доступным сетевым интерфейсам.
    2. Каждые 2 секунды узел отправляет пакет "Hello" на широковещательный адрес 255.255.255.255. Пакет содержит уникальный ID узла, имя пользователя и IP-адрес для приема данных.
    3. Все активные узлы в сети слушают этот порт. При получении пакета "Hello" узел проверяет ID отправителя. Если такой узел еще не известен, он добавляется в локальный список доступных пиров.
    4. Механизм дедупликации предотвращает повторное добавление одного и того же узла.

Этот подход позволяет автоматически находить собеседников без ручной настройки IP-адресов, если устройства находятся в одном широковещательном домене (одной подсети).

### 2. Передача текстовых сообщений

Для текста требуется гарантированная доставка и сохранение порядка пакетов, поэтому используется надежный транспорт.

- **Протокол:** TCP.
- **Порт:** 5001.
- **Алгоритм:**
    1. При отправке сообщения приложение берет список известных пиров из этапа обнаружения.
    2. Для каждого пира устанавливается новое TCP-соединение на порт 5001.
    3. Текст сообщения шифруется симметричным ключом.
    4. Зашифрованные данные сериализуются и отправляются в поток соединения.
    5. Принимающая сторона слушает порт 5001. При входящем соединении она читает поток, десериализует данные, расшифровывает их и добавляет сообщение в локальную базу отображения.
    6. Соединение закрывается сразу после передачи пакета (модель request-response без сохранения постоянного канала).

### 3. Голосовая связь

Голосовой трафик чувствителен к задержкам, но терпим к небольшим потерям пакетов. Поэтому используется быстрый ненадежный транспорт.

- **Протокол:** UDP.
- **Порт:** 5002.
- **Алгоритм:**
    1. Инициация звонка происходит через выбор пира в интерфейсе.
    2. Браузерный движок захватывает аудиопоток с микрофона через MediaDevices API.
    3. Аудио кодируется в чанки (фрагменты) каждые 100 миллисекунд.
    4. Каждый чанк отправляется через UDP-сокет напрямую на IP-адрес и порт 5002 выбранного собеседника.
    5. Принимающая сторона слушает порт 5002, получает бинарные пакеты и воспроизводит их через AudioContext.
    6. В текущей реализации используется прямой обмен. Если прямая видимость отсутствует (разные подсети без маршрутизации UDP), звук передаваться не будет.

### 4. Безопасность и шифрование

Все данные (текст и голос) передаются в зашифрованном виде.

- **Тип шифрования:** Симметричное блочное шифрование (AES-256-GCM аналог в libsodium).
- **Управление ключами:** В текущей версии MVP используется предустановленный общий ключ, зашитый в код приложения. Это позволяет всем участникам сети читать сообщения друг друга без сложной процедуры обмена ключами.
- **Векторы инициализации (Nonce):** Для каждого сообщения генерируется уникальный случайный nonce, что гарантирует криптографическую стойкость даже при передаче одинакового текста.

## Установка и сборка

### Системные требования

- Операционная система: Windows 10/11, Linux, macOS.
- Установленный Rust (последняя стабильная версия).
- Node.js и npm.
- Компилятор C++ (для сборки зависимостей sodiumoxide).

### Процесс сборки

1. Клонируйте репозиторий проекта.
2. Установите зависимости фронтенда командой `npm install`.
3. Запустите режим разработки командой `npm run tauri dev` для тестирования.
4. Для создания релизной версии выполните `npm run tauri build`.
5. Готовые исполняемые файлы будут расположены в директории `src-tauri/target/release/bundle`.

## Конфигурация сети и ограничения

### Работа в одной подсети

В стандартной домашней сети (все устройства подключены к одному роутеру) система работает автоматически. Широковещательные пакеты свободно достигают всех узлов, и пользователи видят друг друга мгновенно.

### Работа в разных подсетях (VLAN)

В крупных сетях (университеты, офисы), где пользователи разделены на разные VLAN, стандартный UDP Broadcast блокируется маршрутизаторами.

**Текущее поведение:**
- Автоматическое обнаружение не сработает. Узлы в подсети A не увидят узлы в подсети B.
- Прямое соединение возможно только если известна точная IP-адресация и настроена маршрутизация между подсетями на уровне сетевого оборудования.

**Пути решения (планируемые):**
1. **Ручной ввод IP:** Добавление поля для явного указания IP-адреса удаленного узла. Это позволит инициировать соединение в обход механизма обнаружения.
2. **UDP Multicast:** Переход с Broadcast на Multicast. Некоторые корпоративные сети пропускают мультикаст-трафик между VLAN, даже блокируя широковещательный.
3. **Режим ретрансляции (Mesh Relay):** Реализация логики, при которой узел, имеющий доступ к двум подсетям, может пересылать пакеты от узлов из одной подсети в другую. Это потребует внедрения счетчика прыжков (TTL) и таблицы хешей полученных пакетов для предотвращения циклов.

## План развития (Roadmap)

### Краткосрочные цели

- **Ручное подключение:** Реализация интерфейса для ввода IP-адреса пира вручную. Это критически важно для работы в сложных сетевых окружениях.
- **Статус доставки:** Визуальная индикация того, что сообщение было успешно получено адресатом.
- **Оптимизация логов:** Система фильтрации и поиска по системным логам для упрощения отладки.

### Среднесрочные цели

- **Настоящее E2E шифрование:** Отказ от общего ключа. Внедрение алгоритма Диффи-Хеллмана для генерации уникального сеансового ключа для каждой пары собеседников. Это обеспечит приватность переписки.
- **Передача файлов:** Возможность отправки изображений и документов через тот же TCP-канал с поддержкой докачки.
- **Улучшение аудиокодека:** Интеграция специализированного аудиокодека (например, Opus) для улучшения качества звука и снижения потребления трафика вместо сырой PCM записи.

### Долгосрочные цели

- **Полноценная Mesh-сеть:** Реализация алгоритмов маршрутизации (Flooding с TTL, векторные дистанции), позволяющих сети работать как единое целое даже при отсутствии прямого контакта между участниками.
- **Мобильные версии:** Портирование на iOS и Android с сохранением единой кодовой базы Rust.
- **Сохранение истории:** Локальная база данных (SQLite) для хранения истории переписки между сеансами работы приложения.
